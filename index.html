🌙 夢境實驗室

整合式睡眠與夢境分析平台｜從需求洞察到可落地系統的完整實踐

專案概述

專案背景：針對現代人睡眠困擾與夢境好奇心，設計一個整合「睡眠記錄」、「夢境解析」與「社群交流」的全方位平台。

開發週期：6 週（2024年10-11月）

團隊規模：4 人（前端 2 人、後端 2 人）

🤝 協作推進與問題解決

在開發過程中，我不只是「做自己的部分」，更多時候是在協調需求、釐清邏輯、推動進度。

  • 前後端需求對齊：主動繪製資料流程圖，確保 API 規格符合前端使用情境
  • 卡關時的溝通協調：當團隊成員遇到技術障礙，協助拆解問題、提出替代方案
  • 使用者視角的檢查點：在每個功能完成後，從使用者角度測試流程，找出潛在問題
  • 文件化設計決策：記錄「為什麼這樣做」的原因，方便後續維護與優化

我的角色與貢獻

🎯 產品定位與需求定義

在專案初期，我主動提出「從使用者痛點出發」的設計思維，協助團隊釐清：我們不只是在做睡眠記錄工具，而是在解決『了解自己睡眠狀況』的需求。

  • 定義核心使用情境：識別出「快速記錄」vs「深度分析」的不同需求，決定功能優先級
  • 設計決策權衡：在「功能豐富度」與「操作簡便性」之間找平衡，確保新手也能快速上手
  • 推動視覺一致性：建立「粉紫 + 留白」設計規範，讓團隊在開發時有明確依據

⚙️ 功能開發與技術決策

負責三大核心模組的開發，過程中不只是「寫功能」，更多是在做「如何讓功能真的好用」的判斷與取捨。

📊 睡眠紀錄系統

關鍵決策：採用「多步驟表單」而非單頁式輸入，降低使用者認知負擔

  • 設計睡眠評分演算法：將主觀的「睡得好不好」轉化為可量化的 A/B/C 分級
  • 規劃檢核邏輯流程：繪製同步/非同步驗證流程圖，避免重複資料進入系統
  • 選擇無限滾動方案：分析使用情境後，判斷「瀏覽歷史紀錄」比「精準搜尋」更重要

📈 每日分析整合

關鍵決策：設計「一鍵查看當日狀態」的概念，降低使用者查找成本

  • 跨模組資料整合：串接睡眠與夢境兩大系統，自動生成每日綜合報告
  • 日曆標記設計：判斷哪些日期有記錄，視覺化呈現使用者的記錄習慣
  • 引導流程優化：根據「是否有夢境」動態調整下一步建議，減少使用者思考時間

⚙️ 管理員系統

關鍵決策：設計「新增後自動導向下一步」的流程，提升管理效率

  • 權限分級機制：區分一般管理員與超級管理員，確保資料安全
  • 意見回饋處理：設計「新增/駁回」雙軌處理方式，讓管理員能快速回應使用者
  • 資料驗證策略：在前端就阻擋無效資料，減少後端負擔與資料庫髒資料

核心邏輯與技術實作

1. 睡眠評分系統設計

分析思維：將主觀的「睡眠品質」轉化為可量化的評分機制

評分維度 權重 邏輯依據
睡眠時長 25-35 分 理想 7-9 小時，過短/過長扣分
睡眠效率 15-30 分 超過 90% 最佳，低於 85% 需改善
睡眠品質 5-25 分 主觀評分 1-5 級轉化
中斷次數 0-10 分 未醒最高分，次數越多扣分越重

分級標準：A 級 (85-100)、B 級 (65-84)、C 級 (45-64)

💡 邏輯思考重點：

設計時考慮「編輯模式」與「新增模式」的差異，編輯時需排除當前紀錄避免誤判重複，透過流程圖釐清各種情境的處理邏輯。

3. 真分頁 vs 無限滾動的選擇

問題分析：睡眠紀錄可能累積大量歷史數據，如何平衡載入速度與使用體驗？

解決方案：

  • 睡眠紀錄：採用「無限滾動」，符合瀏覽歷史記錄的使用習慣
  • 夢境紀錄：採用「真分頁」，支援關鍵字搜尋與精準查找
  • 技術實作：hasMore 標記控制載入、isFetching 鎖定避免重複請求

問題解決與成長

挑戰 1：檢核邏輯的複雜度

問題：涉及輸入錯誤、日期驗證、順序判斷等多層邏輯

解決：繪製流程圖拆解問題，釐清同步/非同步處理順序

學習：將抽象邏輯視覺化，有助於團隊溝通與後續維護

挑戰 2：套件樣式被覆蓋

問題：畫面爆錯、按鈕跑版

解決：查閱官方文件，嘗試 Scoped 樣式避免衝突

學習：理解 CSS 優先級與元件隔離的重要性

挑戰 3：前後端資料不一致

問題：錯誤訊息不明確，一度以為是前端問題

解決：學會從資料端檢查，搭配 console.log 追蹤資料流

學習：建立「從源頭查起」的除錯習慣

💡 跨領域思考的轉變：

從文組「語感思維」轉向「邏輯思維」——不再只是理解內容，而是學會拆解問題、釐清因果關係、設計解決方案。每次 debug 都是一次邏輯訓練。
